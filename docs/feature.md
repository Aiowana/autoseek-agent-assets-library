### 3. 可以改进的细节 (建议)

为了让这份文档更加完美，达到 98+ 分的水平，我建议在以下几个方面进行细化：

#### A. 资产的“合并视图”逻辑 (Union View)
*   **现状**：文档描述了如何查询某个 `ns` 的资产。
*   **问题**：正如你之前提到的，用户通常需要同时看到“系统公共资产”和“用户私人资产”。
*   **改进建议**：在 `4.2 按租户同步` 之后增加一个 **“资产检索逻辑”** 说明。
    *   *规则*：后端在获取资产列表时，应固定查询 `system_global` + 当前租户 `user_{id}`，并在内存中进行合并。

#### B. 资源限流与配额 (Resource Quotas)
*   **问题**：在多租户环境下，如果某个租户的 Git 仓库有 10,000 个资产，可能会撑爆 Redis 内存。
*   **改进建议**：在 `6. 安全设计` 中增加一个 **“配额管理”** 小节。
    *   限制单个租户的最大资产数量（如：每个用户最多 100 个资产）。
    *   限制单个资产的文件大小。

#### C. 同步任务的“并发度”控制
*   **问题**：如果 1,000 个租户同时触发 Webhook，系统会瞬间产生 1,000 个同步任务，可能导致 CPU 爆表。
*   **改进建议**：
    *   引入 **Worker Pool（工作池）** 概念。Webhook 只负责将“任务”丢进 Redis 队列，后台开启固定数量（如 4-8 个）的 Worker 线程/进程按顺序处理同步请求。

#### D. 自动化创建仓库 (Provisioning) 的详细流程
*   **改进建议**：既然你要做“私人 Git”，建议在核心功能里增加一个 **“自动化开通流程 (Tenant Provisioning Flow)”**。
    *   当 `tenants` 表插入新数据且 `is_private=true` 时，系统自动调用 Git API 进行 `Create Repo` -> `Push Initial Template` -> `Create Webhook` 的全流程。

#### E. Redis Key 清理 (GC)
*   **问题**：如果租户注销了，Redis 里的数据如何清理？
*   **改进建议**：增加一个 `tenant_remove` 的清理逻辑说明，使用 `SCAN` 命令匹配前缀批量删除，防止内存溢出。
